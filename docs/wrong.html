Overview

From configuration to movement

The project is intentionally small, which makes it a good study case for understanding how a game loop, object state, and rendering work together.

Configuration

GameConfig stores the rules

The window size, frame rate, square sizes, square count, and maximum speed are collected in one dataclass. That keeps the simulation easy to tune without editing many places.

State

MovingRect stores the behavior

Each square tracks its speed, vector, area, lifetime, and current color. The object carries the data needed to move and draw itself every frame.

Loop

The main loop drives the app

The screen is cleared, the clock advances, objects update, the UI is drawn, and the frame is presented. That loop repeats until the window closes.

Game Loop

How one frame is processed

A frame is not just movement. It is a short pipeline that updates physics, checks interactions, and renders the result.

  1. Read input Poll pygame events and look for the quit event so the program can exit cleanly.
  2. Measure time Call clock.tick(fps) to get the frame time in milliseconds and keep the update rate stable.
  3. Update objects Bounce from walls, move by vector, adjust color, and recalculate chasing behavior.
  4. Render output Draw the squares and the small text overlay, then flip the display buffer.
# simplified frame structure while window_is_open: handle_events() dt = clock.tick(fps) screen.fill(background) for rect in rects: rect = wall_bounce(rect, dt) rect.move_dir(dt) rect.color = lerp_color(end_color, start_color, life_ratio) draw_rect(rect) threat, prey = find_threat_and_prey(rect, rects) rect.vector = run_and_chase_vect(rect, threat, prey, 5) rect.curr_life -= dt draw_ui() display.flip()
Mechanics

Why the squares feel alive

The interesting behavior comes from a few compact functions that combine motion, distance, and lifetime into one simulation.

Motion

Vector-based movement

Each square has a direction vector and a speed scalar. Multiplying by delta time keeps motion tied to real elapsed time instead of frame count.

Interaction

Threat and prey search

The project scans the other squares and picks the nearest larger one as a threat and the nearest smaller one as prey. That gives every square a local decision to make.

Feedback

Color and lifespan

Lifetime fades the color from start to end, and expired squares respawn with a new size and position. The sound cue reinforces that cycle.

  • GameConfig keeps the project settings in one place.
  • MovingRect stores the properties that define a square's behavior.
  • wall_bounce reverses direction when a square reaches the edge of the screen.
  • find_threat_and_prey supports the chase-and-run logic.
  • Try changing square_num. Watch how crowding affects the chase logic.
  • Raise max_speed. Faster motion makes the wall bounce and steering easier to see.
  • Adjust max_life. Shorter lifetimes make the respawn system more obvious.
  • Change the colors. A different palette can make the same logic feel very different.
Next Steps

Ideas for extending the lesson

Use this project as a base for experiments. Small changes here are easy to understand and easy to test.

Experiment

Add keyboard controls

Let the player pause the simulation, spawn more squares, or change the chase intensity at runtime.

Visualize

Draw connection lines

Show which square is being chased and which square is threatening it. That makes the search logic easier to understand.

Refactor

Split the logic into modules

Move configuration, entities, and rendering into separate files once the project grows beyond a single teaching example.

How to run the game
