main.py

Focused on the moving squares pygame simulation. Student-facing guide for structure, behavior, risks, and next steps.

Call Graph
The game starts in main(), initializes pygame, then loops through update, movement, targeting, and drawing work each frame.
Dependency Graph
The simulation depends on pygame plus a small set of stdlib modules, and it also loads the pop sound asset from the media folder.
Startup Sequence
Initialization is linear: create the window, clock, font, sound, and colors before the simulation begins.
Frame Sequence
Each frame polls input, advances time, updates all squares, renders text, and flips the display buffer.
Chase Sequence
The square first finds a larger threat and a smaller prey, then blends those directions into its next vector.
Rects Flow
The active square list shrinks as squares expire, then grows again when respawned squares are appended.
Vector Flow
Direction starts in the constructor, gets nudged by chase and wall logic, and finally drives movement on screen.
Life Flow
Lifetime is the hidden timer behind color fading, death detection, sound playback, and replacement.
Color Flow
Color is derived from remaining life so the visual fade matches the square's age.
3 Good Patterns
✅GameConfig centralizes tuning
The window size, FPS, square count, and size limits live together in one dataclass, which makes the simulation easier to reason about and adjust.
Basics One object holds the main knobs for the game.
Engineering Fewer scattered literals means easier testing and fewer accidental mismatches.
Architecture Configuration stays separate from the animation logic that consumes it.
✅MovingRect bundles state and behavior
Each square carries its own speed, vector, area, life, and color, which keeps the update loop readable.
Basics The square knows the values it needs to move and draw itself.
Engineering Encapsulation makes it simpler to change one entity without rewriting the whole frame loop.
Architecture Object state becomes the unit of simulation instead of a pile of globals.
✅The update loop is explicit
The frame order is easy to follow: events, timing, movement, targeting, UI, and present. That is a strong teaching structure.
Basics The program does one frame of work at a time.
Engineering The control flow stays visible, which helps debugging and experimentation.
Architecture Input, simulation, and rendering are separated by frame boundaries.
2 Potential Issues
⚠️Direction randomization is a no-op
`self.vector.rotate_rad(...)` returns a rotated vector, but the result is never assigned back to `self.vector`, so the direction does not actually change.
Basics The code looks like it changes direction, but the object keeps the old vector.
Engineering This is a correctness bug because the feature is advertised in the method name and docstring.
Architecture Behavior and intent drift apart, which makes the simulation harder to trust.
⚠️Speed randomization also resets to the same value
`randomize_speed()` assigns `self.set_speed()`, which recomputes the same width-based speed instead of introducing any randomness.
Basics The method name suggests variation, but the result is deterministic.
Engineering This makes testing confusing because the branch executes without changing the outcome.
Architecture The object API promises a degree of control that the implementation never delivers.

Type hint coverage is solid
Most public functions are annotated, and the main data structures are typed clearly enough for a beginner-friendly project.

Strengths

  • Core functions such as `main`, `update_screen`, and `find_threat_and_prey` carry return annotations.
  • Container types like `list[MovingRect]` and `tuple[MovingRect | None, MovingRect | None]` are concrete and readable.
  • `GameConfig` and the module-level globals make the simulation state easy to inspect.

Gaps

  • `set_speed`, `set_vector`, and `random_square` do not declare return types even though they are small helpers.
  • `chance` is documented as a float in `randomize_dir` and `randomize_speed`, but the signature still says `int`.
  • Several globals are typed as `None` placeholders, which is workable but less precise than the actual runtime types.

Performance Wins
Simple primitives
The simulation relies on rects, vectors, and a few arithmetic operations. That keeps each object update cheap and easy to follow.
Clear frame pipeline
Timing is handled with `clock.tick(fps)`, which stabilizes the frame loop and avoids busy-waiting.
Performance Risks
Quadratic pair search
`find_threat_and_prey()` runs once per square and scans the whole list each time, so the cost grows quickly as square count increases.
Per-frame surface work
FPS and counter text are re-rendered every frame, and respawns create new square objects plus sound playback. That is fine for small n, but it is the next place to watch.
4 / 6 - Good for the current scale, but the pairwise search will be the first bottleneck.
